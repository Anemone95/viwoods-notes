# Viwoods Notes 二次开发需求

## 工作流

对当前设备前台正在运行的 Viwoods 笔记应用进行二次开发。流程:

1. 通过 `adb` 定位当前前台应用包名,从设备拉取 APK,备份到 `./backup/`。
2. 针对每个功能,先用 **Java 源码** 编写目标逻辑(作为设计稿 + smali patch 参考)。
3. 把 Java 源码编译/对照,生成等价的 **smali 补丁**,对反编译后的 APK 直接打补丁。
4. 重新打包、签名、安装验证。

> 禁止直接改 smali 凭感觉;所有改动必须先有对应的 Java 源文件说明意图和语义。

## 功能需求

### (1) 录音时长上限 10min → 300min

- 现状:应用内录音最长 10 分钟(600 秒 / 600_000 ms)后自动停止。
- 目标:上限改为 300 分钟(18000 秒 / 18_000_000 ms)。
- 实现路径(预期):
  - 搜索常量 `600000`、`10 * 60`、`600` 及相关字段/方法。
  - 定位录音计时器(可能是 `CountDownTimer` / `Handler.postDelayed` / 自定义 Service)。
  - 相关 UI 提示文案("最长 10 分钟" 之类)也需同步改为 300。
- 验收:开始录音,>10min 不中断,能录到接近 300min。

### (2) 长按笔记菜单新增「AI 智能重命名」

- 现状:长按笔记弹出操作菜单,包含 `分享 / 移动 / 重命名 / 加密 / ...`。
- 目标:在菜单中追加一个「AI 智能重命名」入口。
  - 功能:读取当前笔记内容(文字 + OCR 后的手写文本,如果可得),调用设备内置大模型/AI 接口。
  - Prompt:`阅读笔记内容,为笔记本想一个 10 字以内的名字`。
  - 返回结果作为笔记新标题,替换原文件名。
- 实现路径(预期):
  - 定位长按弹窗(BottomSheetDialog / PopupMenu / 自定义 Dialog)构造处。
  - 注入一个新菜单项(图标 + 文案 + 点击回调)。
  - 复用已有「重命名」逻辑写入新标题,差异只在标题来源:本地输入 vs AI 生成。
  - AI 调用:优先调用设备自带 AI SDK(Viwoods 设备可能内置),否则先放占位 stub 留接口。
- 验收:长按笔记 → 新菜单项可见 → 点击后标题被替换为 ≤10 字的合理名称。

### (3) 无边笔记(连续页)

- 现状:笔记按页翻页,到页底需手动新建下一页/翻页;画布 = 一屏,写不下就要换页。
- 目标:新增「无边笔记」模式,**纵向**滑动:
  - 手指**向上滑**(内容上移,视图看到更靠下的部分):
    - 若页面未到底:视图正常向下滚。
    - 若已到底:**动态向下延长当前页高度**,露出新空白,继续在同一页上写。
  - 手指**向下滑**(内容下移,视图回看上面已写内容):视图向上滚;已写内容不缩回/不截断。
  - 不涉及左右 fling,不涉及「+」悬浮提示。
- **实现方案(2026-04-23 决策)**:不采用"单页长到 30×"的字面实现
  —— 笔 SDK 的位图尺寸硬编码读 `DisplayMetrics`,系统笔服务的 EPD
  overlay 物理尺寸固定为屏幕,**从 `com.wisky.notewriter` 一个 APK 内做不
  到把一页画布做得比屏幕高**(详见 `src/feature3_endless_page/architecture.md`
  的 "A 阶段研究结论" 一节)。改走 **Y1 方案:栈式滚动 + 可见分界线**:
  - 一本笔记在 endless 模式下,所有页**竖直栈式**放到一个滚动容器里,
    每一页保持 1× 屏高,页之间画一条水平分界线(Android onDraw 层,
    不参与笔 SDK)。
  - 视觉上用户可以看到"上页下半 + 分界线 + 下页上半"同屏。
  - **写字只发生在"活动页"上**,活动页 = 屏幕中心所在的页。笔 SDK 同一
    时刻只绑定活动页的 bitmap + 屏幕矩形;跨分界线的一笔会被截断。分界
    线本身就是告诉用户"别跨写"。
  - fling 到最后一页底再继续滑 → 静默自动 `createPageToLast()` → 新页
    入栈、可以继续写。不再出现"+"悬浮提示。
- **用户确认的设计参数(2026-04-23 对话,后续多次修订)**:
  - **最终方向(2026-04-23 晚)**:Y1 "栈式滚动 + 分界线" 弃用,改回用户
    最原始的 "真无限画布" 想法 —— 单页可动态向下延伸。B 阶段 spike
    验证了可行性(详见 `src/feature3_endless_page/architecture.md`)。
  - **粒度**:每页独立 endless。一个笔记里的某一页可以长到 N× 屏高,
    其他页仍然 1×。
  - **本期页高上限**:**3 × 屏高**(从最初设想的 30× 降下来)。原因:
    笔 SDK 的 mBitmap 必须 ARGB_8888 才能稳定渲染,尝试过 ALPHA_8 灰度
    不行(渲染挂)。ARGB_8888 3× 约 60 MB 可接受;30× 约 600 MB 设备
    扛不住。
  - **长远优化方向(已搁置)**:**分段缓存方案**。每屏一块独立 ARGB
    bitmap 作为一个 slice,内存里只留当前 ± 1~2 屏,远段 PNG 换出磁
    盘。这样页高可真无上限(受磁盘约束),常驻内存仅 ~60MB。实现复
    杂度约 2-3 天:需要 slice 列表的数据模型、scroll 命中检测、mBitmap
    在 slice 间切换、持久化改造。本期不做,记在这里作为未来扩展点。
  - **PDF 笔记豁免**:底层是 PDF 的笔记永远走传统翻页,不开 endless。
  - **导出**:endless 页导出走现有分页路径(每 1× 屏高切一页输出到
    PDF),不产出超长单页 PDF。
  - **写字 UX**:没有分界线,没有吸附。用户手指滑到底继续向上滑 → 画
    布向下延长 1 屏;停手指停在当前 scrollY。
- **入口(本期先跳过,MVP 期间对所有笔记默认开启)**:后续在笔记
  "..." 菜单加开关项(与 share-audio 同风格)。
- **MVP 验收(本期目标)**:
  - 打开任意笔记,能正常看到内容 + 用笔写字(和传统模式行为一致)。
  - 当 mBitmap 未满 3× 屏高:手指在屏幕上**向上**滑(finger 往上,
    content 也往上 = 相当于把视口往下挪),若当前在画布底 → 画布**向
    下延长 1 屏**;此时用户看到屏幕上出现新的空白区域,可以继续在新区
    域上写字。
  - 已到 3× 上限:再向上滑不再扩,停在当前 scrollY。
  - **向下**滑(finger 往下,content 往下 = 视口往上挪):正常滚动,
    看到前面已写的笔迹。
  - 写到任意位置的笔迹**正确落到 mBitmap 对应 y 行**(通过 scrollY 偏
    移),关闭笔记再进来,全部笔迹保留在原位置。
  - 笔延迟与传统模式无差别。
  - PDF 笔记不触发延长(传统翻页保留)。

## 产物目录结构(规划)

```
notes2/
  backup/                         # 原始 APK 备份
    <pkg>_<versionName>.apk
  work/                           # 反编译 / 重打包工作区
    apktool-out/                  # apktool d 的输出
    repacked.apk
    repacked-signed.apk
  src/                            # 每个功能的 Java 设计稿
    feature1_recording_limit/
    feature2_ai_rename/
    feature3_endless_page/
  patches/                        # 对应 smali 补丁 (.smali 或 diff)
    feature1/
    feature2/
    feature3/
  project.md                      # 本文件
```

## ⚠ 工作区破坏操作

`work/apktool-out/` 是**补丁的工作副本**,补丁就活在里面的 smali/资源文件里。
以下命令会把它连同你的修改一并清掉,**不要在做 feature 期间跑**:

- `make decompile` — 重新跑 `apktool d -f`,覆盖现有 `apktool-out/`。
- `make distclean` — 连同 `jadx-out` 一起删掉整个 `work/`。

正常迭代只需要 `make build` / `make install`,它们不会触发 decompile。

## 风险 & 待确认

- 设备是否具备调用本地 AI 的系统 API?若没有,功能 (2) 需降级为远程调用或交互输入。
- 画布是 Surface 渲染还是 Canvas 渲染,决定无边扩容的实现方式。
- APK 是否有签名校验 / 完整性校验,需要 patch 掉才能安装。
